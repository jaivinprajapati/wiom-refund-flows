Problem

When a partner fails to collect the router, Wiom sometimes arranges a 3PL (third-party logistics) pickup. The 3PL provider picks up the device and delivers it to the warehouse. However, there is currently no system path for Anurag's team to record this pickup and trigger a security deposit refund to the customer.

As a result, customers who had their router collected by a 3PL do not receive their refund — even though the device has been recovered. The resolution depends on manual follow-up with no audit trail.

Solution

Add a 3PL Confirmation section (Section C) to the existing Netbox Return Requests tab in Wiom Hub. Anurag's team uploads pickup proof. This sets a tpl_proof_uploaded flag in the customer DB, which extends the existing refund auto-trigger:

⚡
Extended auto-trigger condition Refund fires when: UPI exists AND (router_recovered OR tpl_proof_uploaded)
Whichever condition arrives second triggers the refund — no manual action by Anurag needed post-upload.

If UPI is not on file, Anurag initiates UPI collection — method depends on whether the customer is active or archived.

🔒
Upload CTA is permission-gated. Only 1–2 team members have access. No CS involvement in this flow.
Flow
1
Anurag searches for customer in Hub
Active customers: searched via Customer tab. Archived customers: searched via Archived Customer tab. This determines the UPI collection method later.
2
Opens Netbox Return Requests tab → sanity check → uploads 3PL pickup proof
Section C upload CTA is disabled if no active PUT exists for this customer. By design, a case only reaches 3PL when a partner has failed to act on a ticket — so a PUT must already exist. This acts as a built-in sanity check. If PUT exists: Anurag uploads PDF/image. Sets tpl_proof_uploaded = true in security_deposit_order table. Stores proof file URL, uploader name, and timestamp against the order.
3a
UPI exists on file → refund auto-fires immediately
Both conditions now met. Juspay payout initiated. Section A shows "Refund initiated" with txn ID. Anurag's job is done.
3b
No UPI → Anurag triggers UPI collection
Active customer: "Send UPI Reminder" button (existing in-app nudge — no change to current functionality).
Archived customer: "Send WhatsApp Link" button is unlocked only after proof is uploaded. Sends tokenised captive portal link.
4
Customer submits UPI → refund auto-fires
Second condition (UPI exists) now met. Juspay payout triggers automatically. No action needed from Anurag.
What's Changing

Hub UI — Netbox Return Requests tab

AreaChange
New: Section C 3PL Confirmation — upload zone (empty state) / proof file display with uploader + timestamp (uploaded state). Upload CTA disabled if no active PUT exists for this customer — sanity check to ensure 3PL is only triggered on valid cases.
Section A — UPI row Archived customers: no "Send UPI Reminder"; instead show "Send WhatsApp Link" (enabled only after proof uploaded). Active customers: existing "Send UPI Reminder" unchanged.
Section A — Refund Status New states: "Awaiting UPI" (proof uploaded, no UPI), "Processing" (auto-initiated), "Completed"
Info bars "Waiting for UPI — refund will auto-initiate on receipt" shown after proof uploaded + no UPI. "Refund auto-initiated" shown on trigger with txn details.

Customer backend

AreaChange
New DB flag + file storage tpl_proof_uploaded, tpl_uploaded_by, tpl_uploaded_at, tpl_proof_file_url — all stored on security_deposit_order table directly.

Why security_deposit_order specifically: Moving forward, refunds may be triggered by other operator-uploaded signals (not just 3PL). Storing all proof against the order table keeps a clean, extensible audit trail — one record per order, all signals in one place.
Refund trigger Extended: fires on UPI AND (router_recovered OR tpl_proof_uploaded)
WhatsApp link API Generate tokenised captive portal link for archived customers. Auto-fire refund on UPI submission via portal.
Out of Scope (V1)
  • CS agent involvement — this is an Ops-only flow
  • Automatic 3PL case detection — Anurag searches manually for now
  • Resend limits or link expiry — follows existing Bhavin nudge behavior
  • Refund amount changes — uses existing calculation in security_deposit_order
  • Other signal types (PUT status, global scanner, device location)
