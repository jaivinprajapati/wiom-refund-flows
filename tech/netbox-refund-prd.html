Overview

When a customer churns and returns their Wiom router (Netbox), a security deposit refund is triggered automatically via the PUT (Pickup Ticket) system. However, there are cases where the device reaches Wiom or a confirmed location but the auto-trigger fails or is not applicable — leaving the customer owed a refund with no system path to process it.

This feature provides ops leads a controlled, audited interface inside Wiom Hub to manually process these refunds, and a browser-based captive portal for archived customers to submit their UPI details.

💡
Two modules in this feature Module 1 — Wiom Hub sub-tab: ops-facing, gated by role, processes the refund.
Module 2 — Captive portal: customer-facing, browser-based, collects UPI from archived customers who no longer have app access.

Context & Problem

Why manual refunds are needed

The automatic refund trigger fires when a PUT is resolved and the device confirmed returned. But it breaks or doesn't apply in the following situations:

  • Device confirmed at partner office via signal/scanner — PUT resolved manually, refund not auto-triggered
  • Device at Wiom warehouse — PUT closed, but customer never submitted UPI and original payment source is unavailable
  • 3PL confirms pickup (via API or email), Anurag verifies — refund not auto-initiated
  • PUT not yet created — customer is asked to start Netbox Return flow in app, but refund is still owed once device is recovered

No-UPI problem

A significant subset of customers never set up UPI in the Wiom app. Once the customer is archived (post-churn), the app is locked and they have no channel to submit UPI. This blocks the refund entirely — we have the device, we owe the money, but nowhere to send it.

Module 1 — Wiom Hub Sub-tab

Entry Points

  • Active customer: User Search → Customers → search by mobile → Customer Details → Netbox Security Refund tab
  • Archived customer: User Search → Archived Customers → search by mobile → Customer Details → Netbox Security Refund tab

The tab is always visible in both contexts. What renders inside changes based on customer status and refund eligibility.

Guard Conditions (checked on tab load)

Before any refund path is shown, two guards are evaluated in order:

Guard 1 Daily limit reached
  • A global daily cap exists on both number of transactions and total Rs amount — whichever hits first blocks the feature for the rest of the day
  • When triggered: show error alert only. No refund CTA. No other UI.
  • Resets at midnight. Ops manager is notified automatically when limit is hit.
  • Limit values: TBD (open item)
Guard 2 Refund already processed
  • Per-customer refund count is capped at 1 — whether triggered by the system (auto) or manually via this portal
  • When triggered: show info alert + read-only refund details card (transaction ID, amount, method, refunded to, status, processed at)
  • No CTA. No further refund possible.

Refund Paths

If both guards pass, the system determines the refund path based on UPI and payment source availability.

PathConditionAction
Path A UPI on file for this customer Show UPI ID, ops verbally confirms with customer, Process Refund CTA
Path B1 No UPI, original payment source available Show masked source + ETA, ops reads to customer, Refund to Source CTA
Path B2 No UPI, no payment source Send refund link via WhatsApp — customer submits UPI via captive portal
ℹ
Path B1 + B2 shown together when source available When the original payment source exists, both options are shown — Refund to Source (Option A) and Send Refund Link (Option B). Option A is the preferred path; Option B is a fallback if the customer insists on receiving to UPI.

Pre-Action Checklist

Before any refund CTA is enabled, the ops lead must complete a two-item checklist:

  1. Remarks — reason for manual refund (free text, minimum 5 characters). Required for audit trail.
  2. Screenshot proof — upload a screenshot confirming the device location / CS conversation. Required before any action.

Both checklist items are shown with visual checkmarks (grey → green on completion). All action CTAs are disabled until both items are complete.

The submitted remarks and proof filename are carried forward and shown on the processing/success screen as a read-only Submission card.

Module 2 — Captive Portal (Archived Customer UPI)

A lightweight browser-based flow that allows an archived customer to verify their identity and submit a UPI ID to receive their refund — without needing access to the Wiom app.

ℹ
Portal ownership — open item Whether this is built inside the existing captive portal or as a new standalone page is pending Cx team decision. OTP infrastructure exists and will be reused.

Full Flow

1
Ops sends link from Hub
Clicks "Send Refund Link" → confirms mobile number → link dispatched via WhatsApp (Gupshup)
2
Customer opens link in browser
Link valid for 30 minutes. Landing screen shows customer name and refund amount.
3
OTP verification
OTP sent to registered mobile. Customer enters 6-digit OTP. Resend available.
4
UPI entry
Customer enters UPI ID. System validates format and resolves bank name (via Razorpay / UPI lookup).
5
Confirmation screen
Shows: UPI ID, resolved bank name, amount, ETA. Customer confirms. Option to edit UPI if wrong.
6
Refund auto-initiates
Customer sees success screen with txn reference. Page becomes inactive. WhatsApp confirmations sent: "Initiated" + "Completed".

Business Rules

RuleDetail
Refund amountSecurity deposit amount — TBD, confirm with Bhavin
Per-customer refund capExactly 1 refund per customer, system-wide (system + manual combined). Once processed, all future attempts are blocked.
Daily global limit# of transactions OR total Rs amount — whichever hits first. Values TBD. Resets midnight. Ops manager alerted.
WA link TTL30 minutes from send. Ops can regenerate from Hub; old link immediately invalidated.
UPI per linkEach link allows exactly 1 UPI submission. Once submitted and refund triggered, link is consumed.
Expired link resolutionCustomer calls back → ops regenerates link → new WhatsApp sent. No auto-expiry notification to customer.
Proof requirementScreenshot upload is mandatory before any refund CTA is enabled. Cannot be bypassed.
Remarks requirementFree text remarks mandatory (≥5 chars). Stored as part of audit record.

Access Control

The sub-tab is visible to all Hub users but gated via a permission constant:

CUSTOMER_MODULE_NETBOX_SECURITY_REFUND

Only users with this permission can take any action. Users without it see the tab but all CTAs are disabled. Exact roles to be granted access: TBD (open item — ops leads and above).
