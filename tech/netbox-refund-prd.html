Netbox Security Refund — Manual Portal
Author Jaivin Prajapati Updated 6 Apr 2026 Status In Review

Overview

A controlled, audited interface for ops leads to manually process Netbox security deposit refunds when the auto-trigger fails or doesn't apply. Two modules: Wiom Hub sub-tab (ops-facing) and Captive portal (customer-facing, browser-based, for archived customers with no app access).

Problem

When auto-refund fails

  • Device confirmed at partner office via signal — PUT resolved manually, refund not triggered
  • Device at warehouse — PUT closed, customer never submitted UPI
  • 3PL confirms pickup (API or email), Anurag verifies — refund not auto-initiated
  • PUT not yet created — customer asked to start Netbox Return in app, but refund still owed once device recovered

No UPI problem

  • Customer never submitted UPI in the Wiom app
  • Once archived, app is locked — no channel to collect UPI
  • Device is with Wiom, refund is owed, but nowhere to send it

Scope — V1

  • Manual refund when customer has submitted UPI
  • Manual refund to original payment source (Juspay)
  • WhatsApp link + captive portal for archived customers to submit UPI
  • Daily limit guardrail + per-customer refund cap
  • Role-gated access
Out of scope V1: active customer UPI submission path (guide customer via app while on call) — separate ticket.

Diagnostic Signals

Before initiating a refund, CS uses a dedicated diagnostic view in Wiom Hub to assess whether device return is confirmed. This view is separate from the refund processing tab — it is read-only and helps the agent decide which refund reason to select.

Prototype: diagnostic-signals.html

Signal Cards

Three signal cards are shown per customer:

Open — PUT raised, not resolved
Resolved — router_recovered — device confirmed returned via PUT
Resolved — other reason — PUT closed for non-recovery reason

Device Status Postgres t_device IN_WAREHOUSE — device received at Wiom warehouse

INSTALLED (different account) — device moved to another customer

3PL confirmation is not a separate signal card. Anurag will add pickup confirmation details manually via the remarks + proof upload field in the refund checklist.

Eligible vs Action-Needed

ScenarioSignalsVerdict

Device at Wiom warehouse t_device.status = IN_WAREHOUSE Refund eligible
Device at partner office t_device.status = IDLE Refund eligible
Device at another customer t_device.status = INSTALLED, user_account_id differs Refund eligible

PUT resolved — other reason is not a refund trigger. If the PUT was closed for a non-recovery reason but the partner still has the device, the onus is on the partner. CS should ask the partner to scan via Global Scanner before proceeding.

Pre-Action Checklist

  • All action CTAs are disabled until both items are complete
  • Reason (dropdown, required) — one must be selected before text input is accepted:
    • PUT resolved — router recovered
    • Device confirmed at Wiom warehouse
    • Device at partner office (IDLE status)
    • Device at another customer
    • Global Scanner — device scanned at partner location
    • 3PL pickup confirmed
    • Others
    • Not included: "PUT resolved — other reason", "PUT expired", "PUT not yet created" — these don't confirm device return and are not valid refund triggers on their own.
  • Remarks (text input, required) — additional context. Min 5 chars. Both reason + remarks stored in audit log.
  • Screenshot proof (required) — confirms device location or CS conversation
  • Each item has a visual checkmark (grey → green on completion)
  • Submitted reason + remarks + proof filename shown on success screen as read-only Submission card

Captive Portal

Browser-based. No app required. For archived customers to verify identity and submit UPI.

Portal ownership — existing portal or new page? Pending alignment with Rajnikant. OTP infra exists and will be reused.

Flow

  1. Ops sends link from Hub → WhatsApp delivered via Clevertap event (dynamic portal link passed as property). Jaivin to create Clevertap WhatsApp campaign.
  2. Customer opens link in browser → sees name + refund amount
  3. OTP verification — delivery channel (SMS or WhatsApp) to be confirmed with Akash
  4. Customer enters UPI → system resolves bank name via lookup
  5. Confirmation screen — customer reviews UPI, amount, ETA → confirms
  6. Refund auto-initiates → success screen + WhatsApp confirmations sent ("initiated" + "completed")

Link TTL: 30 minutes. Ops regenerates from Hub if expired. Old link immediately invalidated.

Expired link resolution: Customer calls back → ops regenerates → new WhatsApp sent.
